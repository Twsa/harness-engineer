                  <h3>Getting consistent results from AI-assisted coding starts before the first line of code</h3>

          <p dir="ltr"><span>Building AI development workflows taught me a key lesson: the AI writes better code when you design the environment it works in, a practice some are calling&nbsp;</span><em>harness engineering</em><span>. The secret is structured context rather than free-form tickets. This is the journey that led me to this approach and the two techniques that worked.</span></p><h2 dir="ltr"><span>The problem: Vague in, vague out</span></h2><p dir="ltr"><span>I was working on a multi-repository project that included a Rust backend, a TypeScript frontend, Helm charts, and performance tests. I wanted to use AI to accelerate the software development lifecycle (SDLC). The first attempts went like this: paste a Jira feature description into an AI coding tool, tell it to "implement this," and hope for the best.</span></p><p dir="ltr"><span>The results were unpredictable. Sometimes the AI would succeed. Other times it would hallucinate file paths, invent APIs that didn't exist, or modify the wrong module entirely. The failure mode was always the same: the AI was guessing about the code base instead of looking at it.</span></p><p dir="ltr"><span>The fundamental issue wasn't the AI's coding ability. Instead, it was the input I provided. A free-form Jira ticket like "add CSV export for SBOMs" leaves the solution space wide open. Which service handles software bill of materials (SBOMs)? What's the existing export pattern? Where do the tests go? Without answers to these questions, the AI fills in the blanks with suboptimal solutions.</span></p><h2 dir="ltr"><span>The fix: A two-phase workflow</span></h2><p dir="ltr"><span>I split the problem into two distinct phases, planning and implementation, which each have their own constraints. Constraining the solution space at each phase made the output more consistent.</span></p><h3 dir="ltr"><span>Phase 1: The repository impact map</span></h3><p dir="ltr"><span>Before any tasks are created, I have the AI scan the actual code base and produce a repository impact map. It's grounded in the repository as the single source of truth.</span></p><p dir="ltr"><span>The AI uses Language Server Protocol (LSP) and Model Context Protocol (MCP) servers to inspect the repository structure: it finds symbols, traces references, and searches for patterns. Then it produces a map like this:</span></p><pre><code class="language-plaintext">trustify (backend):
&nbsp;&nbsp;changes:
&nbsp;&nbsp;&nbsp;&nbsp;- Add CSV serialization for SBOM query results
&nbsp;&nbsp;&nbsp;&nbsp;- Add GET /api/v2/sbom/export endpoint

trustify-ui (frontend):
&nbsp;&nbsp;changes:
&nbsp;&nbsp;&nbsp;&nbsp;- Add "Export CSV" button to SBOM list toolbar</code></pre><p dir="ltr"><span>The important step:&nbsp;</span><em>A human reviews this map before anything else happens</em>.<span> If the AI picked the wrong module, missed a repository, or invented a nonexistent endpoint pattern, I catch it here to prevent bad tasks and poor code.</span></p><p dir="ltr"><span>This checkpoint catches a common class of errors early: structurally incorrect plans that produce code that compiles but solves the wrong problem.</span></p><h3 dir="ltr"><span>Phase 2: The structured task template</span></h3><p dir="ltr"><span>Once the impact map is approved, each unit of work becomes a Jira task that follows a strict template:</span></p><pre><code class="language-plaintext">## Repository
trustify

## Description
Add a CSV export endpoint for SBOM query results.

## Files to Modify
- `modules/sbom/src/service.rs`—add CSV serialization method
- `modules/sbom/src/endpoints.rs`—add GET handler

## Implementation Notes
Follow the existing JSON export pattern in `SbomService::export_json()`.
Reuse the `QueryResult` type from `modules/sbom/src/model.rs`.

## Acceptance Criteria
- [ ] GET /api/v2/sbom/export?format=csv returns valid CSV
- [ ] Existing JSON export still works

## Test Requirements
- [ ] Integration test in `modules/sbom/tests/` following existing test patterns</code></pre><p dir="ltr"><span>Every field is intentional:</span></p><ul><li dir="ltr" aria-level="1"><code><span>Repository</span></code><span> scopes the AI to a single repository to avoid cross-repo confusion.</span></li><li dir="ltr" aria-level="1"><code><span>Description</span></code><span> summarizes what this task refers to.</span></li><li dir="ltr" aria-level="1"><code><span>Files to Modify</span></code><span> lists real paths found during the analysis phase, not guesses.</span></li><li dir="ltr" aria-level="1"><code><span>Implementation Notes</span></code><span> reference actual symbol names and existing patterns. When the AI reads </span><code>Follow the existing JSON export pattern in `SbomService::export_json()`.</code><span>, it can look up that function and mimic its structure.</span></li><li dir="ltr" aria-level="1"><code><span>Acceptance Criteria</span></code><span> give the AI a concrete checklist to verify against.</span></li><li dir="ltr" aria-level="1"><code><span>Test Requirements</span></code><span> provides guidance on the test coverage required for this tasks.</span></li></ul><p dir="ltr"><span>This template is the contract between planning and implementation. The AI implementing the task doesn't need to make architectural decisions. Those constraints are established in the impact map.</span></p><h2 dir="ltr"><span>Why this works</span></h2><p dir="ltr"><span>The underlying principle is simple: The more you constrain the solution space, the more predictable the output becomes.</span></p><p dir="ltr"><span>When the AI plans against real code, it produces grounded plans. When it implements against a structured spec with real file paths and symbol names, it produces targeted code changes.</span></p><p dir="ltr"><span>I iterated on this, tightening the feedback loop each time. Early versions of the planning skill would dump a wall of analysis without a review checkpoint. As a result, the AI would rush from "here's what I found" straight to creating tasks. Adding the explicit pause for human review of the impact map was a small change that reduced errors noticeably.</span></p><p dir="ltr"><span>Similarly, the task template evolved. The first version had a generic&nbsp;Notes field. Renaming it to&nbsp;Implementation Notes and requiring references to actual code symbols changed the results. —Vague descriptions signaled that the planning phase wasn't grounded in real analysis.</span></p><h2 dir="ltr"><span>The takeaway</span></h2><p dir="ltr"><span>If you're building AI-assisted development workflows, don't start by writing code. Start with harness engineering by designing the environment:</span></p><ul><li dir="ltr" aria-level="1">Make the AI look at real code before planning.<span> A repository impact map built from symbol analysis grounds every downstream task in real code.</span></li><li dir="ltr" aria-level="1">Give the AI structured constraints to implement against.<span> File paths, symbol names, existing patterns to follow: the more specific the input, the more consistent the output.</span></li><li dir="ltr" aria-level="1">Put a human review checkpoint between planning and implementation.<span> Catching a wrong assumption in a three-line impact map costs far less than catching it in a pull request.</span></li><li dir="ltr" aria-level="1">Invest in the quality of your Jira features<span>. A well-written feature description with clear scope, concrete acceptance criteria, and relevant context directly improves the plans and code the AI produces. The same&nbsp;</span><em>structure in, structure out</em><span> principle applies to the requirements themselves.</span></li></ul><p dir="ltr"><span>The shift from prompt engineering to harness engineering marks a change in how we treat AI in the development cycle. When you stop treating the AI as a magic box and start treating it as a component within a structured environment, your results become predictable and reproducible. By grounding the AI in your code base and providing clear implementation constraints, you turn a flighty assistant into a reliable contributor.</span></p><p dir="ltr"><span>Structure in, structure out.</span></p><h3><span>Next steps</span></h3><ul><li dir="ltr" aria-level="1">Make the repository the single source of truth, including conventions.<span> Move style guides, naming rules, and architectural decisions into the repo itself (for example,&nbsp;</span><code><span>CLAUDE.md</span></code><span> or&nbsp;</span><code><span>CONVENTIONS.md</span></code><span>). When the AI reads the code base, it picks up the rules automatically—no copy-pasting into prompts.</span></li><li dir="ltr" aria-level="1">Treat the harness as software you maintain.<span> Skills, prompts, and MCP configurations are code. Version them, review them in PRs, and refactor them when they drift. A stale prompt rots just like a stale test.</span></li><li dir="ltr" aria-level="1">Close the feedback loop with the agent.<span> When a generated plan or PR is wrong, trace the mistake back to the input: a missing constraint, a vague acceptance criterion, a symbol the AI couldn't see. Fix the harness, not just the output.</span></li><li dir="ltr" aria-level="1">Expand the agent's toolbox through MCP.<span> Each new MCP integration (such as continuous integration (CI) status, deployment logs, and runtime metrics) gives the AI another source of real data.</span></li></ul>

  var ATTR = 'data-parh-prod-placement-workaround';
  var MAX = 35;
  var DELAY = 50;
  var n = 0;

  function getAnchor() {
    return document.querySelector('#rlc .rhd-universal-header')
      || document.querySelector('.rhd-universal-header');

    var toggle = document.querySelector('.parh-chat-drawer__toggle');
