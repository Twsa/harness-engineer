/>  <div class="inner-block-content rich-content">
    <p data-block-key="r26o2">The growing landscape of AI agent development is overloaded with acronyms: MCP, A2A, UCP, AP2, A2UI, and AG-UI, just to name a few. If you’ve ever looked at this list of protocols and felt like you were staring at a wall of competing standards, you are not alone. To help you understand their value, we are going to demonstrate what each one does to save you from writing and maintaining custom integration code for every single tool, API, and frontend component your agent touches.</p><p data-block-key="3178o">We will put these protocols into practice by using Agent Development Kit (ADK) to build a multi-step supply chain agent for a restaurant. This scenario works well as a test case because ordering wholesale ingredients requires checking inventory databases, communicating with remote supplier agents, executing secure transactions, and rendering interactive dashboards.</p><p data-block-key="21c8n">We&#x27;ll start with a bare LLM that hallucinates everything, then add protocols one by one until it can check real inventory, get specialist quotes, place orders, authorize payments, and render interactive, streaming dashboards.</p><h3 data-block-key="jjj7n" id="1.-model-context-protocol-(mcp)"><b>1. Model Context Protocol (MCP)</b></h3><p data-block-key="2lopd">Let’s look at the first hurdle you hit when building an agent: connecting it to your systems and your data. If your kitchen manager agent needs to check inventory or email a supplier, you would normally have to write custom integration code for that specific API. If a service has a dozen endpoints, you&#x27;re writing and maintaining a dozen custom tools just for that service.</p><p data-block-key="fmfg3">The <a href="https://modelcontextprotocol.io/"><b>Model Context Protocol (MCP)</b></a> eliminates this busywork by giving you a single standard connection pattern for hundreds of servers. Servers advertise their tools, and your agent discovers them automatically. And because the MCP servers are maintained by the teams who built those systems, your agent always gets the latest tool definitions without you writing or updating any integration code.</p>
</div>

    <p data-block-key="r26o2">ADK provides first-class support for this via <a href="https://google.github.io/adk-docs/mcp/">McpToolset</a>. Instead of writing a bunch of API requests from scratch, our kitchen manager can now read from a real PostgreSQL database using the <a href="https://google.github.io/adk-docs/integrations/mcp-toolbox-for-databases/">MCP Toolbox for Databases</a>, look up recipes via the <a href="https://google.github.io/adk-docs/integrations/notion/">Notion MCP</a>, and take action by emailing suppliers using the <a href="https://google.github.io/adk-docs/integrations/mailgun/">Mailgun MCP</a>.</p>
</div>  <div class="inner-block-content code-block line-numbers">
    <pre><code class="language-python">from google.adk.agents import Agent

# 1. Inventory database - MCP Toolbox for Databases (PostgreSQL, SQLite, BigQuery, etc.)

# 2. Kitchen SOPs and recipes - Notion MCP (read menus, ingredient lists, supplier contacts)
notion_tools = McpToolset(connection_params=StdioConnectionParams(
    server_params=StdioServerParameters(
        command=&quot;npx&quot;, args=[&quot;-y&quot;, &quot;@notionhq/notion-mcp-server&quot;],
        env={&quot;NOTION_TOKEN&quot;: NOTION_TOKEN}),
    timeout=30))

# 3. Email suppliers about orders - Mailgun MCP (send confirmations, track delivery)
mailgun_tools = McpToolset(connection_params=StdioConnectionParams(
    server_params=StdioServerParameters(
        command=&quot;npx&quot;, args=[&quot;-y&quot;, &quot;@mailgun/mcp-server&quot;],
        env={&quot;MAILGUN_API_KEY&quot;: MAILGUN_API_KEY}),
    timeout=30))

kitchen_agent = Agent(
    model=&quot;gemini-3-flash-preview&quot;,
    name=&quot;kitchen_manager&quot;,
    instruction=&quot;You manage a restaurant kitchen. Check inventory, look up recipes, email suppliers.&quot;,
    tools=[inventory_tools, notion_tools, mailgun_tools],

    <p data-block-key="r26o2">You can browse the full ecosystem of <a href="https://google.github.io/adk-docs/integrations/?topic=mcp">MCP integrations in ADK</a> to see what&#x27;s available.</p><h3 data-block-key="h85ug" id="2.-agent2agent-protocol-(a2a)"><b>2. Agent2Agent Protocol (A2A)</b></h3><p data-block-key="72hf8">With MCP handling data access, the next challenge is expertise. Your kitchen manager can check inventory, but it doesn&#x27;t know today&#x27;s wholesale prices, supplier quality grades, or delivery windows. That knowledge lives with different remote agents, potentially built by different teams, on different frameworks, running on different servers. In some cases, the raw data might never be exposed by API but could be exposed via an agentic interface. Without a standard protocol, you&#x27;d have to write and maintain custom integration code for each one, and redeploy every time a remote agent changes.</p><p data-block-key="elmbg">The <a href="https://a2a-protocol.org/">Agent2Agent (A2A) protocol</a> standardizes how agents discover and communicate with each other. Each A2A agent publishes an Agent Card at a well-known URL (<code>/.well-known/agent-card.json</code>) that describes its name, capabilities, and endpoint. See the <a href="https://a2a-protocol.org/latest/specification/">A2A protocol docs</a> for the full specification.</p>
</div>

  <div class="inner-block-content rich-content">
    <p data-block-key="r26o2">Our kitchen manager agent fetches these cards to learn what each remote agent does, then routes queries to the right one at runtime. Adding a new remote agent is as simple as adding a new URL, eliminating the need for manual code change or re-deployments.</p><p data-block-key="b2lok">ADK’s <code>RemoteA2aAgent</code> routes to one remote agent per turn. When a query spans multiple remote agents, such as checking price, quality, and delivery at once, you can use the <code>a2a-sdk</code> directly, which is the approach we use here.</p>
</div>  <div class="inner-block-content code-block line-numbers">
    <pre><code class="language-python"># An A2A agent serves an Agent Card at /.well-known/agent-card.json:

#   &quot;name&quot;: &quot;pricing_agent&quot;,
#   &quot;description&quot;: &quot;Checks today&#x27;s wholesale market prices for food items.&quot;,
#   &quot;skills&quot;: [{&quot;id&quot;: &quot;pricing&quot;, &quot;name&quot;: &quot;Price Check&quot;,
#               &quot;description&quot;: &quot;Check current wholesale market prices&quot;}],
#   &quot;url&quot;: &quot;http://pricing-agent:8001/&quot;,
#   &quot;version&quot;: &quot;1.0.0&quot;
# }

# EXPOSE: Turn any ADK agent into an A2A service
from google.adk.a2a.utils.agent_to_a2a import to_a2a
app = to_a2a(pricing_agent, port=8001)

# DISCOVER: Resolve the Agent Card and create a client - just a URL
from a2a.client.client_factory import ClientFactory
client = await ClientFactory.connect(&quot;http://pricing-agent:8001&quot;)
card = await client.get_card()
print(f&quot;{card.name} - {card.description}&quot;)
# -&gt; &quot;pricing_agent - Checks today&#x27;s wholesale market prices for food items.&quot;

# CALL: Send a message via the A2A protocol
from a2a.client.helpers import create_text_message_object
msg = create_text_message_object(content=&quot;What&#x27;s today&#x27;s wholesale price for salmon?&quot;)
async for response in client.send_message(msg):

There&#x27;s no record of what limits were set, which merchants are approved, or when the authorization expires.</p><p data-block-key="428ns">The Agent Payments Protocol (AP2) adds that missing layer with typed mandates that provide non-repudiatable proof of intent and enforce configurable guardrails on every transaction.     <p data-block-key="cvlj4">The agent handles each stage of the request using a different protocol:</p><p data-block-key="2ulf9"><b>Stage 1: Gather information</b></p><ul><li data-block-key="1bj5p">First, check what&#x27;s in stock. <b>MCP</b> queries the inventory database for salmon (<code>check_inventory</code>).</li><li data-block-key="3bhvt">The stock is low. What&#x27;s the going rate? <b>A2A</b> queries the remote pricing and quality agents (<code>ask_agent</code>).</li></ul><p data-block-key="aceuj"><b>Stage 2: Complete the transaction</b></p><ul><li data-block-key="3durm">The price looks right, place the order. <b>UCP</b> sends a checkout request to Example Wholesale (<code>place_order</code>).</li><li data-block-key="d8gho">But who approved this? <b>AP2</b> secures the order with a payment mandate within configured guardrails (<code>authorize_payment</code>).</li></ul><p data-block-key="efdq7"><b>Stage 3: Present the results</b></p><ul><li data-block-key="6979n">Now show the user what happened. <b>A2UI</b> composes interactive widgets from the results.</li><li data-block-key="2k9dc">Stream it all to the frontend. <b>AG-UI</b> delivers the tool calls and text responses in real time.</li></ul>
</div>

    <p data-block-key="fp8g7">Six protocols, each solving a different problem, all working through a single agent.</p><h2 data-block-key="4frx8" id="tips-for-working-with-these-protocols"><b>Tips for working with these protocols</b></h2><p data-block-key="f2a91"><b>Know what problem each protocol solves:</b> MCP connects agents to tools and data. A2A connects agents to other agents. UCP standardizes commerce. AP2 handles payment authorization. A2UI defines what to render. AG-UI defines how to stream it. Understanding these boundaries keeps your architecture clean.</p><p data-block-key="fuf7r"><b>Add protocols as you need them:</b> You don&#x27;t need all six in your agent on day one. Most agents start with MCP for data access. As your requirements grow (multi-agent communication, commerce, payments, rich UI, streaming), bring in the protocol that solves that specific problem.</p><p data-block-key="bmha0"><b>Don’t start from scratch:</b> Before building with a protocol, check for an ADK integration, an official SDK, and sample code. These protocols move fast, and the official tooling handles details you don&#x27;t want to reimplement yourself.</p><p data-block-key="bqmv4"><b>Adopt standards early:</b> These protocols are still maturing, but the patterns they establish (discovery via well-known URLs, typed request/response schemas, standard event streams) make your agent compatible with the growing ecosystem of tools, services, and other agents.</p><h2 data-block-key="wj8d9" id="get-started-now"><b>Get Started Now</b></h2><p data-block-key="b6ggn">Every code sample in this post uses <a href="https://google.github.io/adk-docs/">ADK</a>. Set up your first agent, connect it to an <a href="https://google.github.io/adk-docs/tools-custom/mcp-tools/">MCP</a> server, and start adding protocols from there. Browse the <a href="https://google.github.io/adk-docs/integrations/">ADK Integrations</a> to see what tools and services are already available. Links to each protocol&#x27;s documentation and samples are throughout this guide.</p><p data-block-key="fpcfr">We can’t wait to see what you build.</p>
