    <section class="gh-content gh-canvas">
        <p>There's a hidden mechanic in Claude Code that most people don't think about until it costs them an hour of work. Compaction is the automatic compression of your conversation context when you approach the token limit. I ran into it when my always-on agent "forgot" a critical rule from the beginning of a session after 200 minutes of continuous operation. Here's what I learned by digging into how it works.</p><h2 id="what-is-compaction-and-why-it-exists">What is compaction and why it exists</h2><p>Claude Code runs in your terminal and maintains long sessions. Unlike ChatGPT where each message is a separate request, Claude Code accumulates context: files it read, commands it executed, grep results, diffs, error messages. All of this stacks into one massive prompt.</p><p>The problem: models have context window limits. Claude Sonnet has 200K tokens, Opus goes up to 1M. But even a million tokens runs out when your agent is actively working through a codebase. Compaction kicks in when context reaches roughly 95% of the limit.</p><p>What happens: Claude Code takes the entire conversation history, sends it to a separate model call with a prompt like "compress this into key facts," and replaces the full history with a condensed summary. You don't see this happening — at some point, the agent starts working from "memory" of past interactions rather than the full transcript.</p><h2 id="what-survives-compaction-what-doesnt">What survives compaction, what doesn't</h2><p>From my experience running agents daily, compaction reliably preserves:</p><ul><li>The current task and its immediate context</li><li>File names that were recently modified</li><li>Recent errors and their solutions</li><li>General project architecture</li></ul><p>What consistently gets lost:</p><ul><li>Instructions from the start of the session — "don't touch this file," "use this format"</li><li>Intermediate decisions — why you chose approach A over B</li><li>Specific code snippets discussed 50 messages ago</li><li>Subtle style rules — "no emoji," "no Co-Authored-By in commits"</li></ul><p>This makes sense: compaction optimizes for "what to do next," not "why we did what we did." Decision context is the first casualty of compression.</p><h2 id="custom-compaction-prompts">Custom compaction prompts</h2><p>Claude Code lets you override the compaction prompt. This might be the most underrated setting. In <code>settings.json</code> (at <code>~/.claude/settings.json</code>), there's a parameter:</p><pre><code>{
  "compactPrompt": "Preserve ALL rules from CLAUDE.md verbatim. Keep file paths, error messages, and architectural decisions. Summarize tool outputs but keep their conclusions."
}</code></pre><p>I've tested multiple variations. The default prompt is a generic "summarize the conversation." It works fine for short sessions, but for agents running for hours, you need more precision.</p><p>My current compaction prompt:</p><pre><code>{
  "compactPrompt": "When compacting, you MUST preserve:\n1. All rules and constraints from CLAUDE.md and system prompt — copy them verbatim\n2. Current task context and progress\n3. File paths that were modified\n4. Specific error messages and their solutions\n5. User corrections and preferences stated during the session\n\nYou MAY summarize:\n- Tool call outputs (keep conclusions, drop raw output)\n- File contents that were read (keep what was learned, drop the text)\n- Exploratory steps that led nowhere"
}</code></pre><p>The difference is noticeable. After switching to this custom prompt, my agents stopped "forgetting" CLAUDE.md rules during long sessions.</p><h2 id="claudemd-vs-compaction-where-to-put-what">CLAUDE.md vs compaction: where to put what</h2><p>The key insight from three months of running Claude Code agents: <strong>never rely on compaction for critical rules</strong>. Everything the agent must always remember should live in CLAUDE.md.</p><p>CLAUDE.md loads as part of the system prompt at the start of every session. Compaction doesn't touch it — it exists outside the conversation history. It's the only place that's guaranteed to survive any compression.</p><p>My strategy:</p><ul><li><strong>CLAUDE.md</strong> — architectural rules, code style, prohibitions, credential paths, naming conventions. Anything that applies to every session</li><li><strong>Conversation context</strong> — current task, intermediate results, discussions. It's OK to lose this during compaction</li><li><strong>Project files</strong> — TODO.md, ARCHITECTURE.md, notes. The agent can re-read these if compaction erased the context</li></ul><p>Rule of thumb: if you catch yourself repeating the same instruction to the agent — move it to CLAUDE.md. I covered CLAUDE.md configuration in detail in my <a href="https://okhlopkov.com/claude-code-setup-mcp-hooks-skills-2026/">Claude Code setup guide</a>.</p><h2 id="how-to-tell-when-compaction-happened">How to tell when compaction happened</h2><p>Claude Code shows a <code>[context compacted]</code> indicator in the terminal. But if you're multitasking, it's easy to miss. A more reliable signal: watch the cost counter in the bottom-right corner. A sudden cost spike (compaction is a separate API call) plus a context counter reset means compaction just fired.</p><p>Another telltale sign: the agent starts re-asking things you already discussed. "Which file has the config?" — even though you edited that file 20 minutes ago. That's a clear signal that context was compressed and details were lost.</p><h2 id="when-compaction-causes-real-problems">When compaction causes real problems</h2><p>A concrete case from my work. I run a <a href="https://okhlopkov.com/en-ai-agents-practice-claude-code-voice-commands/">server-agent</a> — Claude Code running 24/7 on a VPS, accessible via Telegram. The agent receives a message, processes it, waits for the next one. Sessions can last hours.</p><p>The problem: after compaction, the agent would "forget" access control rules. I have a permission matrix — which files can be read from which chat context. These rules were in the initial prompt, but after compression the model lost them and started responding to requests it should have blocked.</p><p>The fix: I moved all security rules into CLAUDE.local.md (which loads on top of the base CLAUDE.md). Now they're in the system prompt, not in the conversation history — compaction can't touch them.</p><h2 id="practical-tips-for-managing-compaction">Practical tips for managing compaction</h2><p><strong>Customize your compactPrompt.</strong> Tailor it to your workflow. Working with data? Ask it to preserve SQL queries. Writing articles? Ask it to keep style rules. The default is too generic for specialized work.</p><p><strong>Keep sessions short when possible.</strong> A fresh session means clean context plus full CLAUDE.md. Sometimes that's better than a long session with two compaction events. I've found that breaking work into focused 30-60 minute sessions produces better results than marathon 4-hour runs.</p><p><strong>Use files as external memory.</strong> Ask the agent to write key decisions to a TODO.md or NOTES.md file in your project. After compaction, it can re-read them. This pattern is especially important for <a href="https://okhlopkov.com/second-brain-obsidian-claude-code/">Obsidian-based workflows</a> where the vault acts as long-term memory.</p><p><strong>Put everything critical in CLAUDE.md.</strong> Rules that must always apply go there, nowhere else. Don't hope that compaction will preserve them — assume it won't.</p><p><strong>Test compaction intentionally.</strong> Run a long session, wait for compression, then check — does the agent still remember the rules from the beginning? This will save you hours of debugging later.</p><p>Compaction isn't a bug — it's a necessary compromise. Context windows are growing, but as long as they're finite, compression will be part of working with AI agents. Better to understand how it works than to be surprised when your agent "forgets" a critical rule at the 200-minute mark.</p><hr><p>Follow me on <a href="https://x.com/danokhlopkov?ref=okhlopkov.com">Twitter/X @danokhlopkov</a> for more on AI agents and automation.</p>
