			<h1 class="entry-title"><div>KernelEvolve: How Meta’s Ranking Engineer Agent Optimizes AI Infrastructure</div></h1>		</div><!-- .entry-meta -->

	</header><!-- .entry-header -->

			<figure id="post-feat-image-container">
				<img width="1999" height="1116" src="https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-Hero.png" class="wp-post-image" alt="" />						</figure>
		
	<div class="entry-authors-social">
					<div class="entry-authors">
				By <a href="https://engineering.fb.com/author/gang-liao/" title="Posts by Gang Liao" class="author url fn" rel="author">Gang Liao</a>, <a href="https://engineering.fb.com/author/yavuz-yetim/" title="Posts by Yavuz Yetim" class="author url fn" rel="author">Yavuz Yetim</a>, <a href="https://engineering.fb.com/author/ruichao-xiao/" title="Posts by Ruichao Xiao" class="author url fn" rel="author">Ruichao Xiao</a>, <a href="https://engineering.fb.com/author/zewei-jiang/" title="Posts by Zewei Jiang" class="author url fn" rel="author">Zewei Jiang</a>, <a href="https://engineering.fb.com/author/raghav-boinepalli/" title="Posts by Raghav Boinepalli" class="author url fn" rel="author">Raghav Boinepalli</a>, <a href="https://engineering.fb.com/author/sheela-yadawad/" title="Posts by Sheela Yadawad" class="author url fn" rel="author">Sheela Yadawad</a>, <a href="https://engineering.fb.com/author/liyuan-li/" title="Posts by Liyuan Li" class="author url fn" rel="author">Liyuan Li</a>, <a href="https://engineering.fb.com/author/nathan-yan/" title="Posts by Nathan Yan" class="author url fn" rel="author">Nathan Yan</a>, <a href="https://engineering.fb.com/author/ajit-mathews/" title="Posts by Ajit Mathews" class="author url fn" rel="author">Ajit Mathews</a>, <a href="https://engineering.fb.com/author/chunqiang-cq-tang/" title="Posts by Chunqiang (CQ) Tang" class="author url fn" rel="author">Chunqiang (CQ) Tang</a>, <a href="https://engineering.fb.com/author/carole-jean-wu/" title="Posts by Carole-Jean Wu" class="author url fn" rel="author">Carole-Jean Wu</a>, <a href="https://engineering.fb.com/author/gaoxiang-liu/" title="Posts by Gaoxiang Liu" class="author url fn" rel="author">Gaoxiang Liu</a>			</div>

	<div class="entry-content">

		<p><i><span style="font-weight: 400;">This is the second post in the Ranking Engineer Agent blog series exploring the autonomous AI capabilities accelerating Meta&#8217;s Ads Ranking innovation. The </span></i><i><span style="font-weight: 400;">previous post</span></i><i><span style="font-weight: 400;"> introduced <a href="https://engineering.fb.com/2026/03/17/developer-tools/ranking-engineer-agent-rea-autonomous-ai-system-accelerating-meta-ads-ranking-innovation/" target="_blank" rel="noopener">Ranking Engineer Agent&#8217;s ML exploration capability</a>, which autonomously designs, executes, and analyzes ranking model experiments. This post covers how to optimize the low-level infrastructure that makes those models run efficiently at scale. We introduce KernelEvolve, an agentic kernel authoring system used by Ranking Engineer Agent and generally applicable to a range of AI models beyond Ads Ranking.</span></i></p>
<h2>Summary</h2>
<ul>
<li style="font-weight: 400;" aria-level="1"><span style="font-weight: 400;">Meta operates a large fleet of heterogeneous hardware — NVIDIA GPUs, AMD GPUs, Meta&#8217;s custom MTIA silicon chips, and CPUs. Using this hardware effectively and efficiently requires developing </span><span style="font-weight: 400;">software that translates high-level model operations into efficient, chip-specific instructions called </span><span style="font-weight: 400;">optimized kernels. Authoring and optimizing kernels must be done for each new chip generation and ML model architecture. Beyond standard kernel operators like </span><span style="font-weight: 400;">general matrix multiplications (GEMMs)</span><span style="font-weight: 400;"> and convolutions covered by vendor libraries, production workloads require many custom operators across ranking models. With the number of models and number of hardware types and generations, hand-tuning by kernel experts doesn&#8217;t scale.</span></li>
<li style="font-weight: 400;" aria-level="1"><span style="font-weight: 400;">To address the volume of performance optimization work required by the increasing number of models X number of hardware types &amp; generations, we built </span><a href="https://arxiv.org/abs/2512.23236?ref=engineeringatmeta" target="_blank" rel="noopener"><span style="font-weight: 400;">KernelEvolve</span></a><span style="font-weight: 400;">, an agent to optimize performance used by </span><a href="https://engineering.fb.com/2026/03/17/developer-tools/ranking-engineer-agent-rea-autonomous-ai-system-accelerating-meta-ads-ranking-innovation/" target="_blank" rel="noopener"><span style="font-weight: 400;">Meta’s Ranking Engineer Agent</span></a><span style="font-weight: 400;">. It enables: </span>
<ul>
<li style="font-weight: 400;" aria-level="2"><b>Faster development</b><span style="font-weight: 400;">: Compresses weeks of expert engineering time optimizing kernels, including profiling, optimizing, and cross-hardware debugging, into hours of automated search and evaluation, freeing engineers for other work.</span></li>
<li style="font-weight: 400;" aria-level="2"><b>Better  performance</b><span style="font-weight: 400;">: Over 60% inference throughput improvement for the </span><a href="https://engineering.fb.com/2024/12/02/production-engineering/meta-andromeda-advantage-automation-next-gen-personalized-ads-retrieval-engine/"><span style="font-weight: 400;">Andromeda</span></a><span style="font-weight: 400;"> Ads model on NVIDIA GPUs and over 25% training throughput improvement for an ads model on Meta&#8217;s custom MTIA silicon chips.</span></li>
<li style="font-weight: 400;" aria-level="2"><b>Broad applicability</b><span style="font-weight: 400;">: Optimizes across public and proprietary hardware including NVIDIA GPUs, AMD GPUs, </span><a href="https://ai.meta.com/blog/meta-mtia-scale-ai-chips-for-billions/"><span style="font-weight: 400;">MTIA</span></a><span style="font-weight: 400;"> chips and CPUs</span><span style="font-weight: 400;">, generating kernels in high-level DSLs like Triton, Cute DSL, and FlyDSL, as well as low-level languages including CUDA, HIP, and MTIA C++.</span></li>
</ul>
</li>
<li style="font-weight: 400;" aria-level="1"><span style="font-weight: 400;">KernelEvolve treats kernel optimization as a search problem: a purpose-built job-harness evaluates each candidate kernel, feeds diagnostics back to the LLM, and drives a continuous search over hundreds of alternatives, exceeding the performance of human expert generated kernels.</span></li>
<li style="font-weight: 400;" aria-level="1"><span style="font-weight: 400;">More details are available in the paper, “</span><a href="https://arxiv.org/abs/2512.23236?ref=engineeringatmeta" target="_blank" rel="noopener"><span style="font-weight: 400;">KernelEvolve: Scaling Agentic Kernel Coding for Heterogeneous AI Accelerators at Meta</span></a><span style="font-weight: 400;">,” which will appear at the </span><a href="https://iscaconf.org/isca2026/" target="_blank" rel="noopener"><span style="font-weight: 400;">53rd International Symposium on Computer Architecture (ISCA) 2026</span></a><span style="font-weight: 400;">.</span></li>
</ul>
<hr />
<p><span style="font-weight: 400;">Every day, Meta serves billions of AI-powered experiences, from personalized recommendation to generative AI assistants, on a global infrastructure including diverse hardware from NVIDIA, AMD, and Meta’s custom MTIA silicon chips. Behind every training or inference request lies a layer of highly optimized low-level hardware kernels:  small programs that translate high-level model operations into instructions a specific chip can execute efficiently. As AI models grow more complex and the hardware landscape diversifies, the number of kernels scales across hardware platforms, model architectures and operator types, resulting in thousands of configurations that can no longer realistically be tuned by human experts, creating a critical bottleneck that delays hardware enablement and performance tuning and slowing model iteration cycles that drive critical advances in ML technology and its applications.     </span></p>
<p><span style="font-weight: 400;">Today, we are sharing </span><a href="https://arxiv.org/abs/2512.23236?ref=engineeringatmeta" target="_blank" rel="noopener"><span style="font-weight: 400;">KernelEvolve</span></a><span style="font-weight: 400;">, an agentic AI system that improved ads model inference throughput by 60% in hours of experimentation, a task that would take human experts weeks. KernelEvolve autonomously generates and optimizes production-grade kernels for heterogeneous hardware used in training and inference, including NVIDIA GPUs, AMD GPUs, Meta&#8217;s custom MTIA silicon, and CPUs. Unlike typical large language model (LLM)-based agents that perform one-shot code generation, KernelEvolve treats kernel optimization as a search problem. It explores hundreds of alternative kernel implementations to identify a solution that often matches or exceeds human expert performance, and does so in hours instead of weeks. </span><span style="font-weight: 400;">In Meta&#8217;s production environment, KernelEvolve is optimizing code that serves trillions of daily inference requests.</span><span style="font-weight: 400;"><br />
</span><span style="font-weight: 400;"><br />
</span><span style="font-weight: 400;">KernelEvolve represents a fundamental shift in how we think about the relationship between AI software and hardware. Where kernel development was once a manual, expert-driven process that struggled to keep pace with hardware and model evolution, KernelEvolve makes it continuous and automated — adapting as each changes. As Meta continues to diversify its AI hardware portfolio, the ability to rapidly generate optimized kernels for new chips  substantially reduces the engineering effort required to integrate heterogeneous hardware for training and inference. </span></p>
<h2><span style="font-weight: 400;">The Challenge: The Bottleneck of Explosive Kernel Growth</span></h2>
<p><span style="font-weight: 400;">We’re seeing explosive kernel growth because the total number of kernels scales with the product of three factors: {hardware types and generations X model architectures X number of operators}. This product results in thousands of unique kernel configurations that must be written, tested, and maintained. Hand-tuning each kernel doesn’t scale, and kernel experts alone can’t keep up with the pace.</span></p>
<h3><span style="font-weight: 400;">Hardware Heterogeneity</span></h3>
<p><span style="font-weight: 400;">Meta&#8217;s accelerator fleet now spans NVIDIA GPUs, AMD GPUs, and Meta&#8217;s custom </span><a href="https://ai.meta.com/blog/next-generation-meta-training-inference-accelerator-AI-MTIA/"><span style="font-weight: 400;">MTIA</span></a><span style="font-weight: 400;"> silicon, each with fundamentally different memory architectures and hierarchies, instruction sets, and execution models. A kernel that runs optimally on one platform may perform poorly or fail entirely on another. And the complexity doesn&#8217;t stop at vendor boundaries. Even within a single hardware family, successive generations introduce architectural changes that require different optimization strategies. Meta&#8217;s MTIA roadmap spans four chip generations in two years (</span><a href="https://ai.meta.com/blog/meta-mtia-scale-ai-chips-for-billions/"><span style="font-weight: 400;">MTIA 300 through 500</span></a><span style="font-weight: 400;">), each introducing new compute capabilities, memory bandwidth characteristics, and numeric data types optimized for evolving workloads. A kernel optimized for one generation will underperform when run on the next generation of the same hardware architecture.</span></p>
<h3><span style="font-weight: 400;">Model Architecture Variation</span></h3>
<p><span style="font-weight: 400;">Meta&#8217;s recommendation models have evolved through three major phases: from early embedding-based deep learning recommendation models, to </span><a href="https://engineering.fb.com/2024/11/19/data-infrastructure/sequence-learning-personalized-ads-recommendations/" target="_blank" rel="noopener"><span style="font-weight: 400;">sequence learning models</span></a><span style="font-weight: 400;"> that process engagement histories with attention mechanisms, to Meta’s </span><a href="https://engineering.fb.com/2025/11/10/ml-applications/metas-generative-ads-model-gem-the-central-brain-accelerating-ads-recommendation-ai-innovation/" target="_blank" rel="noopener"><span style="font-weight: 400;">Generative Ads Recommendation Model (GEM</span></a><span style="font-weight: 400;">), and most recently Meta&#8217;s </span><span style="font-weight: 400;">foundation inference model that brings LLM-scale to ads (</span><a href="https://engineering.fb.com/2026/03/31/ml-applications/meta-adaptive-ranking-model-bending-the-inference-scaling-curve-to-serve-llm-scale-models-for-ads/" target="_blank" rel="noopener"><span style="font-weight: 400;">Meta Adaptive Ranking Model</span></a><span style="font-weight: 400;">)</span><span style="font-weight: 400;">. Each generation introduces operator types the previous generation never needed. Beyond these generational shifts, Meta&#8217;s production stack simultaneously serves fundamentally different model families, each with its own unique operators, and a single ads request may traverse multiple families in one serving call. With a vast and growing number of distinct models in production, every new architecture extends the matrix of operators that must be optimized across hardware.</span></p>
<h3><span style="font-weight: 400;">Kernel Diversity Beyond Standard Libraries</span></h3>
<p><span style="font-weight: 400;">Vendor libraries like cuBLAS and cuDNN cover a set of common operations — GEMMs, convolutions, standard activations — but even these standard operators resist one-size-fits-all solutions. A single operator like matrix multiplication behaves differently across contexts: The optimal kernel for a training batch differs from an inference serving request, and tensor shapes vary widely across ranking stages and ranking models, creating a combinatorial space of configurations that neither human experts nor today&#8217;s compiler-based autotuning and fusion can fully cover at scale. Beyond standard operators, production workloads are dominated by a long tail of operators that fall outside library coverage. These include data preprocessing transforms like feature hashing, bucketing, and sequence truncation that prepare raw input for model inference, as well as custom model operators like fused feature interaction layers and specialized attention variants that are unique to Meta&#8217;s architectures. </span></p>
<p><span style="font-weight: 400;">None of these custom operators appear in vendor libraries, and many are too workload-specific to warrant a library implementation. Without native accelerator implementations, these operators either fall back to CPU — forcing disaggregated serving architectures with significant latency overhead — or run via unoptimized code paths that underutilize hardware. </span></p>
<p><span style="font-weight: 400;">The problem compounds with hardware diversity. A hand-tuned NVIDIA kernel cannot simply be recompiled for AMD GPUs or MTIA. Each new model architecture extends the tail further, and each new chip multiplies the work required to cover it. </span></p>
<h2><span style="font-weight: 400;">How KernelEvolve Addresses These Challenges</span></h2>
<p><span style="font-weight: 400;">Each challenge maps to a specific architectural decision:</span></p>

<h2><span style="font-weight: 400;"><br />
KernelEvolve: Searching for Optimal Kernels</span></h2>
<p><span style="font-weight: 400;">KernelEvolve approaches this challenge differently from standard AI coding assistants. Rather than prompting an LLM to generate a single kernel and testing it, the system formalizes kernel optimization as a structured search problem across the space of possible implementations. Under the hood, a purpose-built long-running job harness drives each iteration – compiling candidates, evaluating correctness and performance, profiling hardware utilization, and generating analysis reports – all while handling the multi-minute build cycles and infrastructure failures that make native approaches impractical.</span></p>
<figure id="attachment_23821" aria-describedby="caption-attachment-23821" style="width: 1634px" class="wp-caption alignnone"><img decoding="async" class="size-full wp-image-23821" src="https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png" alt="" width="1634" height="1488" srcset="https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png 1634w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png?resize=916,834 916w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png?resize=768,699 768w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png?resize=1024,933 1024w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png?resize=1536,1399 1536w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png?resize=96,87 96w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image1.png?resize=192,175 192w" sizes="(max-width: 992px) 100vw, 62vw" /><figcaption id="caption-attachment-23821" class="wp-caption-text">Figure 1: ​​How a kernel optimization request flows through KernelEvolve&#8217;s six components.</figcaption></figure>
<h3><span style="font-weight: 400;">LLM Synthesizer</span></h3>
<p><span style="font-weight: 400;">An LLM generates candidate kernels across multiple programming languages and hardware targets — from high-level DSLs like Triton, </span><a href="https://github.com/facebookexperimental/triton"><span style="font-weight: 400;">TLX</span></a><span style="font-weight: 400;">, CuTe DSL, and FlyDSL, to low-level backends including CUDA, HIP, and MTIA C++.</span><span style="font-weight: 400;"><br />
</span><span style="font-weight: 400;"><br />
</span><span style="font-weight: 400;">Rather than using static prompts, the synthesizer constructs </span><b>dynamic, context-aware prompts</b><span style="font-weight: 400;"> that are continuously enriched with runtime diagnostics, hardware constraints, and the historical signals from prior candidate optimization evaluation. This replaces the traditional approach of maintaining separate prompt templates for debugging, performance tuning, and correctness verification with a single adaptive interface that unifies these workflows into a single adaptive interface that drives a continuous, feedback-driven optimization loop.</span></p>
<h3><span style="font-weight: 400;">Tree Search Engine</span></h3>
<p><span style="font-weight: 400;">The system explores the optimization space using graph-based search algorithms, including Monte Carlo tree search and evolutionary strategies. Each kernel candidate becomes a node in a search tree. The engine selects promising candidates, applies transformations, evaluates results, and decides whether to explore further or backtrack — balancing exploitation of known-good strategies against exploration of novel approaches.</span></p>
<p><span style="font-weight: 400;">Crucially, nodes do not evolve in isolation. Each node carries a configurable memory operator that determines how it draws context from the search tree when generating the next round of candidates. A node may inherit its parent’s optimization trajectory to refine a promising direction, compare against siblings to learn what differentiates high-performing variants, combine insights from both parent and sibling histories, or start with a clean slate to escape local optima. This selective memory mechanism allows the tree search to move beyond simple independent sampling – sibling nodes collaborate by surfacing complementary strategies, parent-child chains preserve and deepen successful optimization paths, and memory-free restarts inject diversity when the search stagnates.</span></p>
<figure id="attachment_23822" aria-describedby="caption-attachment-23822" style="width: 1999px" class="wp-caption alignnone"><img decoding="async" class="size-full wp-image-23822" src="https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png" alt="" width="1999" height="1538" srcset="https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png 1999w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png?resize=916,705 916w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png?resize=768,591 768w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png?resize=1024,788 1024w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png?resize=1536,1182 1536w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png?resize=96,74 96w, https://engineering.fb.com/wp-content/uploads/2026/04/Meta-KernalEvolve-REA-image2.png?resize=192,148 192w" sizes="(max-width: 992px) 100vw, 62vw" /><figcaption id="caption-attachment-23822" class="wp-caption-text">Figure 2: How the tree search engine navigates the optimization space to find high-performing kernels.</figcaption></figure>
<h3><span style="font-weight: 400;">Retrieval-Augmented Knowledge Base</span></h3>
<p><span style="font-weight: 400;">To generate optimized code for hardware the underlying LLM was never trained on, KernelEvolve maintains a hierarchical knowledge base organized into three categories: correctness constraints that enforce valid kernel implementations, platform-agnostic optimization guidance covering debugging and tuning strategies, and hardware-specific documentation containing architectural details for each accelerator platform. The system retrieves relevant knowledge dynamically based on runtime signals. For example, a memory bandwidth bottleneck triggers retrieval of memory hierarchy documentation; a compilation error activates debugging guidance. </span></p>
<p><span style="font-weight: 400;">This knowledge base is not static. As the system solves new optimization problems it distills successful strategies into reusable skills — compact optimization patterns and debugging heuristics — that are continuously written back into the knowledge base. This self-evolving skill library acts as a form of </span><b>in-context reinforcement learning</b><span style="font-weight: 400;">: Each successful exploration enriches the context available to future sessions, enabling the system to solve similar problems faster and with fewer search steps, without requiring model retraining.</span></p>
<h3><span style="font-weight: 400;">Automated Evaluation Framework</span></h3>
<p><span style="font-weight: 400;">Every generated kernel passes through a rigorous validation pipeline that checks both correctness — bitwise accuracy against reference implementations — and performance. And evaluation goes far beyond a single runtime number.</span></p>
<p><span style="font-weight: 400;">KernelEvolve leverages a stack of profiling tools, each targeting a different level of analysis. </span><a href="https://github.com/meta-pytorch/tritonbench"><span style="font-weight: 400;">TritonBench</span></a><span style="font-weight: 400;"> validates numerical correctness against PyTorch baselines and measures end-to-end speedup across production input shapes. PyTorch Profiler captures system-level execution timelines, including kernel launch overhead and host-device synchronization. For GPU targets, tools like </span><a href="https://developer.nvidia.com/nsight-compute"><span style="font-weight: 400;">NCU</span></a><span style="font-weight: 400;"> provide kernel-level hardware metrics — occupancy, memory throughput, instruction mix — while </span><a href="https://ieeexplore.ieee.org/document/11395207"><span style="font-weight: 400;">Proton</span></a><span style="font-weight: 400;"> delivers intra-kernel instruction-level latency and pipeline behavior. For MTIA targets, MTIA Insight provides comprehensive accelerator-specific instrumentation: PE utilization, fixed-function engine metrics (DPE, SFU, MLU utilization and stall cycles), cache behavior, and per-PE memory bandwidth counters.</span></p>
<p><span style="font-weight: 400;">Rather than treating these tools as standalone steps, KernelEvolve unifies them through a compiler-centric abstraction. The framework composes analysis through job graphs: compiler transforms insert MLIR-level instrumentation, profiling passes collect metrics, and trace synthesis produces structured output. This means the search engine doesn&#8217;t just see &#8220;kernel A is 1.2x faster than kernel B&#8221; — it sees why: whether the bottleneck is memory-bound, compute-bound, or limited by occupancy — and feeds that diagnostic signal back into the LLM synthesizer to guide the next round of candidates.</span></p>
<h3><span style="font-weight: 400;">Shared Data Foundation</span></h3>
<p><span style="font-weight: 400;">Every optimization session contributes to a shared data foundation. When one engineer&#8217;s exploration discovers an effective tiling strategy for a class of operators, that insight becomes available to every future session targeting similar workloads — creating a compounding effect where the system grows more capable with each use. Early adopters perform the hardest exploration; subsequent users inherit much closer to optimal starting points and refine from there. </span></p>
<h3><span style="font-weight: 400;">Agentic Reinforcement Learning</span></h3>
<p><span style="font-weight: 400;">Every optimization session generates structured training data as a natural byproduct: agentic trajectories capturing the reasoning, code transformations, and evaluation feedback behind high-performing kernels. </span><b>This domain-specific data is rare and valuable</b><span style="font-weight: 400;">. It encodes optimization intuition that no public dataset contains. </span></p>
<p><span style="font-weight: 400;">We use this data to post-train smaller, specialized models through agentic reinforcement learning, where the reward signal comes directly from measured kernel performance. The result is a virtuous cycle where better models produce better kernels in fewer reasoning tokens and fewer search steps, which in turn generate higher-quality training data. Over successive iterations, this compounding flywheel enables us to self-host increasingly efficient models that are compact enough to run cost-effectively at scale while retaining the optimization capability of much larger frontier models. </span></p>
<h2><span style="font-weight: 400;">Enabling Proprietary AI Chips</span></h2>
<p><span style="font-weight: 400;">One of the most consequential capabilities of this architecture is its ability to generate optimized code for hardware that does not exist in any public training dataset. </span></p>
<p><span style="font-weight: 400;">Meta&#8217;s custom MTIA chips present a unique programming challenge. Because these chips are proprietary, no public LLM has been trained on MTIA code. A standard coding assistant lacks the context to write optimized MTIA kernels because it has never seen MTIA documentation, instruction set details, or programming idioms. </span></p>
<p><span style="font-weight: 400;">KernelEvolve solves this through systematic knowledge injection. We encode MTIA-specific documentation (architecture manuals, instruction set references, memory hierarchy specifications, and optimization patterns) directly into the retrieval-augmented knowledge base. When the system targets MTIA, it retrieves and incorporates this proprietary knowledge into its reasoning, effectively &#8220;learning&#8221; the hardware in real time. </span></p>
<p><span style="font-weight: 400;">This approach extends to any new accelerator. When a new chip arrives, the engineering cost shifts from writing thousands of kernels by hand to curating a set of hardware documents and injecting them into the knowledge base. The system then autonomously generates optimized kernels for the new platform, ensuring the software stack is ready at the speed of hardware deployment rather than the speed of manual engineering.</span></p>
<h2><span style="font-weight: 400;">KernelEvolve’s Impact Across Benchmark and Production</span></h2>
<p><span style="font-weight: 400;">KernelEvolve has delivered strong results across both standardized benchmarks and production workloads.</span></p>
<p><b>Benchmark performance</b><span style="font-weight: 400;">: On KernelBench, a benchmark suite of 250 kernel optimization problems from Stanford spanning three difficulty levels, KernelEvolve achieves a </span><b>100%</b> <b>pass</b> <b>rate</b><span style="font-weight: 400;"> — all generated kernels are both functionally correct and faster than their PyTorch reference implementations. The system also validates 160 PyTorch ATen operators with 100% correctness across three hardware platforms (480 total configurations).</span></p>
<p><b>Production</b> <b>speedups</b><span style="font-weight: 400;">: On Meta&#8217;s MTIA chips, KernelEvolve’s generated kernels, which spanned compute-bound, memory-bound, and custom operations, achieved speed ups of over 25% training throughput improvement on an ads model. On NVIDIA GPUs, it delivered more than 60% inference throughput improvement over a model with highly optimized kernels including torch.compile and vendor libraries — performance gains that directly translate to serving capacity and infrastructure efficiency.</span></p>
<p><b>Hardware</b> <b>coverage</b><span style="font-weight: 400;">: The system generates optimized kernels for NVIDIA GPUs, AMD GPUs, Meta&#8217;s custom MTIA silicon, and CPUs — from a single unified framework. Rather than maintaining separate prompt templates per platform, the system dynamically retrieves hardware-specific constraints and optimization patterns, adapting to each target through retrieval augmentation rather than manual prompt engineering.</span></p>
<h3><span style="font-weight: 400;">Development Velocity</span></h3>
<p><span style="font-weight: 400;">Kernel development that previously required weeks of expert effort — profiling, iterating on tiling strategies, debugging edge cases across hardware — now completes in hours through automated search and evaluation. This shifts engineer time from writing low-level code to higher-value work such as designing model architectures, improving training techniques, and defining optimization objectives.</span></p>
<h2><span style="font-weight: 400;">How It All Fits Together</span></h2>
<p><span style="font-weight: 400;">An engineer specifies a target operator, hardware platform, and performance goals. The system then autonomously:</span></p>
<ol>
<li style="font-weight: 400;" aria-level="1"><b>Retrieves</b><span style="font-weight: 400;"> relevant hardware documentation and optimization knowledge from the knowledge base. </span></li>
<li style="font-weight: 400;" aria-level="1"><b>Generates</b><span style="font-weight: 400;"> an initial set of kernel candidates using the LLM synthesizer with context-aware prompting. </span></li>
<li style="font-weight: 400;" aria-level="1"><b>Evaluates</b><span style="font-weight: 400;"> each candidate for correctness and performance using distributed benchmarking infrastructure. </span></li>
<li style="font-weight: 400;" aria-level="1"><b>Feeds results</b><span style="font-weight: 400;"> back into the search engine, which selects the most promising candidates and applies further optimizations. </span></li>
<li style="font-weight: 400;" aria-level="1"><b>Iterates</b><span style="font-weight: 400;"> steps 1-4, exploring the search tree until the termination criteria are met — either a performance target is achieved, the search budget is exhausted, or progress stalls. </span></li>
<li style="font-weight: 400;" aria-level="1"><b>Outputs</b><span style="font-weight: 400;"> the best-performing, fully validated kernel, ready for production deployment.</span></li>
</ol>
<p><span style="font-weight: 400;">The process runs on Meta&#8217;s distributed infrastructure, evaluating thousands of candidates in parallel. Persistent storage of search trees and implementations lets the system build on prior results when targeting new model </span><span style="font-weight: 400;">variants</span><span style="font-weight: 400;"> or hardware generations.</span></p>
<h2><span style="font-weight: 400;">Looking Ahead</span></h2>
<p><span style="font-weight: 400;">The same agentic techniques powering KernelEvolve — structured reasoning, retrieval-augmented knowledge, closed-loop evaluation — can be applied to hybrid model search, compiler optimization, memory management, and system configuration. </span><span style="font-weight: 400;">KernelEvolve</span><span style="font-weight: 400;"> represents an early step toward the vision of a </span><b>Ranking Engineer Agent</b> <b>that can continuously optimize its own performance-critical infrastructure.</b></p>
<p><span style="font-weight: 400;">Within REA, </span><a href="https://engineering.fb.com/2026/03/17/developer-tools/ranking-engineer-agent-rea-autonomous-ai-system-accelerating-meta-ads-ranking-innovation/"><span style="font-weight: 400;">ML Exploration</span></a><span style="font-weight: 400;"> discovers better models. </span><span style="font-weight: 400;">KernelEvolve makes them production-ready. Together, they accelerate how quickly ranking improvements reach advertisers.</span><span style="font-weight: 400;"> </span></p>
